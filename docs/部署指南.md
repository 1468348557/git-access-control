# GitLab 部署指南

## 架构概览

门禁代码集中在 `zh-1087/hobo-git-gate` 仓库维护，业务项目通过 `ci_config_path` 指向该仓库的 `.gitlab-ci.yml`，无需在每个业务项目中放置门禁文件。

```
业务项目 A              业务项目 B              业务项目 C
ci_config_path ──────► zh-1087/hobo-git-gate@main ◄────── ci_config_path
                             │
                   .gitlab-ci.yml
                   scripts/git_gate.sh
                   scripts/approval.sh
```

CI 运行时，job 先从门禁仓库 clone 脚本到 `/tmp/gate`，再执行。工作目录仍是业务项目代码，不影响 git 操作。

---

## 1. 前置条件

### 1.1 门禁仓库访问权限

门禁仓库为 Private，需要在门禁仓库创建 Deploy Token 供业务项目 clone 使用：

1. 门禁仓库 → Settings → Repository → Deploy tokens
2. 创建 token，scope 选 `read_repository`，名称任意如 `ci-gate-read`
3. 记录生成的 token 值（Username 和 Token）

### 1.2 业务项目 Runner

每个业务项目需要注册一个 Shell Runner，tag 为 `shell`。

```bash
gitlab-runner register \
  --url http://192.168.8.143:8080/ \
  --registration-token <YOUR_REGISTRATION_TOKEN> \
  --executor shell \
  --tag-list shell \
  --description "shell-runner" \
  --non-interactive
```

---

## 2. 业务项目配置

### 2.1 设置 CI/CD 配置文件路径

每个业务项目只需配置一个值：**Settings → CI/CD → General pipelines → CI/CD configuration file**

```
template/.gitlab-ci.yml@zh-1087/hobo-git-gate@main
```

也可以通过 API 批量设置：

```bash
curl --header "PRIVATE-TOKEN: $TOKEN" \
  -X PUT "https://gitlab.spdb.com/api/v4/projects/$PROJECT_ID" \
  -H "Content-Type: application/json" \
  -d '{"ci_config_path": "template/.gitlab-ci.yml@zh-1087/hobo-git-gate@main"}'
```

### 2.2 配置 CI/CD 变量

**Settings → CI/CD → Variables**，添加以下变量：

| 变量名 | 说明 |
|---|---|
| `GITLAB_PRIVATE_TOKEN` | Personal Access Token，scope 选 `api`，用于 ≤100 行的自动合并 |
| `GATE_DEPLOY_USER` | 门禁仓库 Deploy Token 的 Username（如 `gitlab+deploy-token-123`） |
| `GATE_DEPLOY_PASS` | 门禁仓库 Deploy Token 的 Token 值 |

### 2.3 分支保护配置

**Settings → Repository → Protected branches**：

| 分支 | 保护策略 |
|---|---|
| `master` | 不允许直接 push，仅允许 MR 合并 |
| `uat`, `uat_*` | 不允许直接 push，仅允许 MR 合并 |

**Settings → Merge requests → Merge checks**：开启 "Pipelines must succeed"，确保门禁不过不能合入。

---

## 3. 业务项目目录结构（不需要门禁文件）

配置完成后，业务项目**无需包含任何门禁文件**，仓库结构保持原样：

```
业务项目/
├── src/                # 业务代码
├── ...                 # 其他业务文件
└── （无需 .gitlab-ci.yml、scripts 等门禁文件）
```

---

## 4. 更新门禁规则

修改门禁脚本后，只需推送到 `zh-1087/hobo-git-gate` 的 `main` 分支。所有业务项目的下次 MR 自动使用最新脚本，无需逐个更新。

---

## 5. 流水线工作原理

### Stage 1: `git_gate`（自动执行）

推送触发时仅检查分支命名，MR 触发时执行完整五项检查：

| 检查项 | Push | MR |
|---|---|---|
| 分支命名 | ✅ | ✅ |
| 合并方向 | — | ✅ |
| 变更行数 | — | ✅ |
| MR 标题 | — | ✅ |
| 基线校验 | — | ✅ |

变更行数 ≤ 100 行的 MR 由 `git_gate.sh` 直接调用 GitLab API 自动合并。> 100 行的 MR 写入 `approval_status` 文件作为 artifact 传递给 Stage 2。

### Stage 2: `approval`（手动触发）

仅 ≤ 100 行 MR 时出现，需人工点击执行。审批通过后由管理员在 GitLab 手动点击 Merge。

## 6. 验证

```bash
# 合规
git checkout -b FIX-20260525-0001
git push origin FIX-20260525-0001

# 不合规（会被拦截）
git checkout -b fix-bug
git push origin fix-bug
```

创建 MR 时观察流水线是否正常从门禁仓库拉取脚本执行。
