# GitLab 部署指南

## 1. 仓库结构要求

将以下文件放入项目仓库 `scripts/` 目录：

```
./
├── .gitlab-ci.yml          # CI 流水线定义
├── .githooks/
│   └── pre-push             # 本地分支命名拦截
├── Makefile                 # 一键安装 hook
└── scripts/
    ├── git_gate.sh          # 门禁检查脚本
    └── approval.sh          # 人工审批脚本
```

## 2. 配置 GitLab Runner

项目需要注册一个 Shell Runner，tag 为 `shell`.

```bash
sudo gitlab-runner register \
  --url http://gitlab.spdb.com/ \
  --registration-token $REGISTRATION_TOKEN \
  --executor shell \
  --tag-list shell
```

## 3. 本地 pre-push hook（可选）

clone 后执行一次，之后每次 `git push` 自动校验分支命名：

```bash
make install
```

等效于：

```bash
chmod +x .githooks/pre-push
git config core.hooksPath .githooks
```

开发者 push 不合法分支名时，终端直接报错拒绝推送，无需等 CI 跑完。

---

## 4. 流水线工作原理

### Stage 1: `git_gate`（自动执行）

推送触发时仅检查分支命名，MR 触发时执行完整四项检查：

| 检查项 | Push | MR |
|---|---|---|
| 分支命名 | ✅ | ✅ |
| 合并方向 | — | ✅ |
| 变更行数 | — | ✅ |
| 基线校验 | — | ✅ |

变更行数 > 50 行的 MR 不会失败，而是写入 `approval_status` 文件作为 artifact 传递给 Stage 2.

### Stage 2: `approval`（手动触发）

仅 MR 时出现，需人工点击执行。读取 `approval_status` 文件：

- `PASS` — 直接通过
- `NEEDS_APPROVAL:<行数>` — 审批人确认后通过

## 5. GitLab 分支保护配置

在 GitLab 项目的 **Settings → Repository → Protected branches** 中：

| 分支 | 保护策略 |
|---|---|
| `master` | 不允许直接 push，仅允许 MR 合并 |
| `uat`, `uat_*` | 不允许直接 push，仅允许 MR 合并 |
| 其他分支 | 允许直接 push |

同时建议开启 **Settings → Merge requests → Merge checks** 中的 "Pipelines must succeed"，确保门禁不过不能合入。

## 6. 验证

推送一个符合规则的分支和不合规的分支，观察流水线输出：

```bash
# 合规
git checkout -b FIX-20260513-0001
git push origin FIX-20260513-0001

# 不合规（会被拦截）
git checkout -b fix-bug
git push origin fix-bug
```

创建 MR 时，目标分支选 uat/hotfix/release/master 观察合并方向校验结果。
